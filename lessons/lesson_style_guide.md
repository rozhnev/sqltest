# Рекомендации по стилю и оформлению уроков

Этот файл содержит единые рекомендации по структуре, стилю, оформлению и подаче материала для следующих уроков курса. Его цель — сохранить единый визуальный и методический подход во всех языковых версиях.

## Общий принцип

Каждый урок должен быть:

- понятным для начинающего или среднего уровня;
- практически ориентированным;
- компактным по абзацам, но достаточно подробным по смыслу;
- написанным в спокойном, объясняющем, профессиональном тоне;
- полезным для аналитических и прикладных SQL-задач.

Материал должен не просто давать определение, а показывать, **зачем тема нужна**, **как она работает** и **когда применять ее на практике**.

---

## Рекомендуемая структура урока

### 1. Короткий вводный абзац перед заголовком

Урок желательно начинать с 1 короткого абзаца до основного заголовка `#`.

В этом абзаце нужно:

- кратко назвать тему;
- объяснить, что именно изучит читатель;
- обозначить практическую пользу темы;
- завершить абзац ожидаемым результатом обучения.

Пример логики:

"Этот урок знакомит с ... Вы узнаете ... Мы рассмотрим ... К концу урока вы сможете ..."

### 2. Заголовок H1

Основной заголовок оформляется так:

```md
# Урок X.Y: Название темы
```

Название должно быть коротким, точным и совпадать с основной темой урока.

### 3. Контекстный ввод после H1

Сразу после H1 желательно дать 1-2 абзаца контекста:

- связь с предыдущим уроком;
- место темы в общей картине SQL;
- зачем эта тема нужна в реальной работе.

### 4. Основные тематические блоки H2

Урок должен быть разбит на логичные секции с заголовками `##`.

Часто полезны такие типы блоков:

- базовое определение или идея;
- синтаксис;
- примеры;
- сравнение с похожими сущностями;
- практические рекомендации;
- пример из практики;
- ключевые выводы.

### 5. Подразделы H3

Если внутри секции есть несколько отдельных аспектов, используйте `###`.

Это особенно полезно для:

- разбора различий;
- отдельных правил;
- пошаговых примеров;
- перечисления сценариев использования.

### 6. Финальный блок с выводами

В конце урока обязательно должен быть блок:

```md
**Ключевые выводы этого урока:**
```

После него идет список из 4-6 коротких пунктов, которые фиксируют главное.

### 7. Переход к следующему уроку

Последнее предложение должно мягко подводить к следующей теме.

Формат:

- "В следующем уроке мы рассмотрим ..."
- "Далее мы узнаем ..."

Это помогает сохранить ощущение последовательного курса.

## Стиль текста

### Тон

Тон уроков должен быть:

- объясняющим;
- уверенным;
- нейтрально-профессиональным;
- без разговорной избыточности;
- без маркетинговых формулировок.

Подходящий стиль:

- "Теперь рассмотрим..."
- "Это полезно, когда..."
- "На практике такой подход удобен, если..."

Нежелательный стиль:

- слишком рекламный;
- слишком академичный и сухой;
- перегруженный длинными абзацами;
- излишне неформальный.

### Абзацы

Предпочтительны короткие абзацы по 2-4 строки.

Если мысль можно разделить, лучше сделать два коротких абзаца, чем один длинный. Это улучшает читаемость, особенно в техническом материале.

### Объяснение терминов

Новые термины нужно вводить мягко:

- сначала простое объяснение;
- затем точный термин;
- затем пример применения.

Если используется англоязычный термин SQL, его лучше сохранять в исходном виде: `SELECT`, `JOIN`, `CTE`, `TEMPORARY TABLE`.

## Оформление примеров SQL

### Блоки кода

Все SQL-примеры нужно оформлять так:

```sql
SELECT column_name
FROM table_name;
```

Рекомендуется:

- использовать отдельные строки для `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `ORDER BY`;
- выравнивать код аккуратно и предсказуемо;
- не перегружать один пример слишком длинной логикой, если ее можно разбить.

### Примеры должны быть практическими

Примеры лучше строить на понятных таблицах проекта и типичных сценариях. Предпочтительно использовать данные из Sakila:

- `customer`
- `payment`
- `rental`
- `film`
- `category`
- связанные таблицы вроде `inventory`, `film_category`

Пример должен отвечать на вопрос: зачем нужен этот SQL в реальной задаче.

### Комментарий после примера

После важных SQL-примеров полезно давать короткое пояснение:

- что делает запрос;
- что получится в результате;
- почему это полезно.

Иногда удобно использовать строку вида:

```md
*Результат: ...*
```

или

```md
*Примечание: ...*
```

## Визуальное оформление

### Разделители

Между крупными смысловыми блоками желательно использовать:

```md
---
```

Это делает структуру урока визуально более чистой и предсказуемой.

### Списки

Для правил, рекомендаций, отличий и выводов используйте маркированные списки.

Списки должны быть:

- короткими;
- логично сгруппированными;
- без перегрузки длинными предложениями.

### Акценты

Для смысловых акцентов используйте полужирное выделение умеренно:

- для ключевых терминов;
- для важных различий;
- для предупреждений;
- для итоговых выводов.

Не стоит делать слишком много полужирных фрагментов в одном абзаце.

## Содержательные рекомендации

### Что важно включать почти в каждый урок

Если тема позволяет, в уроке желательно раскрывать:

- что это такое;
- где используется;
- как выглядит базовый синтаксис;
- чем отличается от похожих конструкций;
- типичные ошибки или ограничения;
- пример из практики;
- краткий итог.

### Что особенно полезно для курса по SQL-анализу

Так как курс ориентирован на прикладной SQL, материал должен чаще подчеркивать:

- аналитические сценарии;
- промежуточную подготовку данных;
- удобство чтения сложных запросов;
- повторное использование логики;
- надежность и понятность SQL-кода.

### Чего лучше избегать

Не стоит:

- делать урок слишком теоретическим без SQL-примеров;
- использовать слишком абстрактные таблицы без контекста;
- перегружать один урок большим количеством несвязанных нюансов;
- писать чрезмерно длинные вступления;
- перескакивать между темами без логических переходов.

## Рекомендации для перевода на другие языки

При переводе уроков на English, Français и Português важно сохранять:

- ту же структуру разделов;
- тот же порядок примеров;
- тот же уровень детализации;
- ту же практическую направленность;
- те же SQL-блоки без изменения логики.

Можно адаптировать:

- естественные формулировки под язык;
- названия разделов;
- стилистические связки между абзацами.

Не нужно менять:

- смысл примеров;
- структуру урока;
- уровень сложности;
- последовательность объяснения.
