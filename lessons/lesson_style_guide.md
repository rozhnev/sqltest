# Рекомендации по стилю и оформлению уроков

Этот файл содержит единые рекомендации по структуре, стилю, оформлению и подаче материала для следующих уроков курса. Его цель — сохранить единый визуальный и методический подход во всех языковых версиях.

## Общий принцип

Каждый урок должен быть:

- понятным для начинающего или среднего уровня;
- практически ориентированным;
- компактным по абзацам, но достаточно подробным по смыслу;
- написанным в спокойном, объясняющем, профессиональном тоне;
- полезным для аналитических и прикладных SQL-задач.

Материал должен не просто давать определение, а показывать, **зачем тема нужна**, **как она работает** и **когда применять ее на практике**.

## Рекомендуемая структура урока

### 1. Короткий вводный абзац перед заголовком

Урок желательно начинать с 1 короткого абзаца до основного заголовка `#`.

В этом абзаце нужно:

- кратко назвать тему;
- объяснить, что именно изучит читатель;
- обозначить практическую пользу темы;
- завершить абзац ожидаемым результатом обучения.

Пример логики:

"Этот урок знакомит с ... Вы узнаете ... Мы рассмотрим ... К концу урока вы сможете ..."

### 2. Заголовок H1

Основной заголовок оформляется так:

```md
# Урок X.Y: Название темы
```

Название должно быть коротким, точным и совпадать с основной темой урока.

### 3. Контекстный ввод после H1

Сразу после H1 желательно дать 1-2 абзаца контекста:

- связь с предыдущим уроком;
- место темы в общей картине SQL;
- зачем эта тема нужна в реальной работе.

### 3.1. Иллюстрация урока

Для каждого нового урока желательно добавлять отдельную SVG-иллюстрацию в текущем визуальном стиле курса.

Что важно соблюдать:

- иллюстрация должна создаваться в формате `SVG`;
- файл следует сохранять в папку `/images/lessons/`;
- имя файла должно соответствовать уроку, например `lesson9_4-sql-view.svg` или `lesson9_3-temporary-tables.svg`;
- иллюстрацию желательно размещать в уроке после вводного контекста и перед первым основным разделом `##`;
- для вставки в markdown используйте HTML-тег такого вида:

```md
<img src="/images/lessons/lessonX_Y-topic.svg" alt="Lesson illustration" width="100%">
```

Иллюстрация должна быть не декоративной случайностью, а поддерживать тему урока: визуально показывать идею запроса, структуру данных, поток обработки или ключевое отличие рассматриваемого инструмента.

### 4. Основные тематические блоки H2

Урок должен быть разбит на логичные секции с заголовками `##`.

Часто полезны такие типы блоков:

- базовое определение или идея;
- синтаксис;
- примеры;
- сравнение с похожими сущностями;
- практические рекомендации;
- пример из практики;
- ключевые выводы.

### 5. Подразделы H3

Если внутри секции есть несколько отдельных аспектов, используйте `###`.

Это особенно полезно для:

- разбора различий;
- отдельных правил;
- пошаговых примеров;
- перечисления сценариев использования.

### 6. Финальный блок с выводами

В конце урока обязательно должен быть блок:

```md
**Ключевые выводы этого урока:**
```

После него идет список из 4-6 коротких пунктов, которые фиксируют главное.

### 7. Переход к следующему уроку

Последнее предложение должно мягко подводить к следующей теме.

Формат:

- "В следующем уроке мы рассмотрим ..."
- "Далее мы узнаем ..."

Это помогает сохранить ощущение последовательного курса.

## Стиль текста

### Тон

Тон уроков должен быть:

- объясняющим;
- уверенным;
- нейтрально-профессиональным;
- без разговорной избыточности;
- без маркетинговых формулировок.

Подходящий стиль:

- "Теперь рассмотрим..."
- "Это полезно, когда..."
- "На практике такой подход удобен, если..."

Нежелательный стиль:

- слишком рекламный;
- слишком академичный и сухой;
- перегруженный длинными абзацами;
- излишне неформальный.

### Абзацы

Предпочтительны короткие абзацы по 2-4 строки.

Если мысль можно разделить, лучше сделать два коротких абзаца, чем один длинный. Это улучшает читаемость, особенно в техническом материале.

### Объяснение терминов

Новые термины нужно вводить мягко:

- сначала простое объяснение;
- затем точный термин;
- затем пример применения.

Если используется англоязычный термин SQL, его лучше сохранять в исходном виде: `SELECT`, `JOIN`, `CTE`, `TEMPORARY TABLE`.

## Оформление примеров SQL

### Блоки кода

Все SQL-примеры нужно оформлять так:

```sql
SELECT column_name
FROM table_name;
```

Рекомендуется:

- использовать отдельные строки для `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `ORDER BY`;
- выравнивать код аккуратно и предсказуемо;
- не перегружать один пример слишком длинной логикой, если ее можно разбить.

### Примеры должны быть практическими

Примеры лучше строить на понятных таблицах проекта и типичных сценариях. Предпочтительно использовать данные из Sakila:

- `customer`
- `payment`
- `rental`
- `film`
- `category`
- связанные таблицы вроде `inventory`, `film_category`

Пример должен отвечать на вопрос: зачем нужен этот SQL в реальной задаче.

### Комментарий после примера

После важных SQL-примеров полезно давать короткое пояснение:

- что делает запрос;
- что получится в результате;
- почему это полезно.

Иногда удобно использовать строку вида:

```md
*Результат: ...*
```

или

```md
*Примечание: ...*
```

## Визуальное оформление

### Разделители

Между крупными смысловыми блоками желательно использовать:

```md
---
```

Это делает структуру урока визуально более чистой и предсказуемой.

### Иллюстрации

Для новых уроков рекомендуется использовать единый стиль SVG-обложек, который уже применяется в актуальных уроках раздела 9.

Желательные признаки этого стиля:

- формат холста примерно `1200 x 630`;
- светлый мягкий фон с градиентом и простыми декоративными формами;
- крупная темная левая панель с ключевой темой урока;
- справа — 2-4 светлые карточки или схемы, объясняющие идею визуально;
- ограниченное количество текста внутри изображения;
- акцентные цвета для ключевых SQL-объектов или понятий;
- чистая, современная, техническая подача без перегруженности деталями.

Если создается серия уроков одной темы, визуальный язык иллюстраций желательно сохранять единым: одинаковая композиционная логика, близкая палитра, схожие размеры заголовков и карточек.

### Списки

Для правил, рекомендаций, отличий и выводов используйте маркированные списки.

Списки должны быть:

- короткими;
- логично сгруппированными;
- без перегрузки длинными предложениями.

### Акценты

Для смысловых акцентов используйте полужирное выделение умеренно:

- для ключевых терминов;
- для важных различий;
- для предупреждений;
- для итоговых выводов.

Не стоит делать слишком много полужирных фрагментов в одном абзаце.

## Содержательные рекомендации

### Что важно включать почти в каждый урок

Если тема позволяет, в уроке желательно раскрывать:

- что это такое;
- где используется;
- как выглядит базовый синтаксис;
- чем отличается от похожих конструкций;
- типичные ошибки или ограничения;
- пример из практики;
- краткий итог.

### Что особенно полезно для курса по SQL-анализу

Так как курс ориентирован на прикладной SQL, материал должен чаще подчеркивать:

- аналитические сценарии;
- промежуточную подготовку данных;
- удобство чтения сложных запросов;
- повторное использование логики;
- надежность и понятность SQL-кода.

### Чего лучше избегать

Не стоит:

- делать урок слишком теоретическим без SQL-примеров;
- использовать слишком абстрактные таблицы без контекста;
- перегружать один урок большим количеством несвязанных нюансов;
- писать чрезмерно длинные вступления;
- перескакивать между темами без логических переходов.

## SEO-оптимизация уроков

Следующие правила применяются прежде всего к английской версии (`.en.md`), но желательно соблюдать их и в других языковых версиях.

### Frontmatter с мета-тегами

Каждый урок должен начинаться с YAML-блока frontmatter:

```md
---
title: "Keyword-rich Page Title — up to 60 characters"
description: "Direct answer to the user's question in 120–155 characters. What they will learn and why it matters."
keywords: ["primary keyword", "secondary keyword", "long-tail variant"]
teaches: ["learning outcome 1", "learning outcome 2", "learning outcome 3"]
about: ["Entity 1", "Entity 2", "Entity 3"]
---
```

- `title` — используется в `<title>` страницы; должен содержать главный поисковый запрос темы.
- `description` — **обязательное поле**. Используется в `<meta description>` и в `LearningResource.description` schema.org. Правила написания:
  - длина: **120–155 символов** (оптимально для отображения в выдаче);
  - начинается с **прямого ответа** на вопрос темы, а не с маркетинговой фразы;
  - содержит 1–2 ключевых слова темы урока;
  - отвечает на вопрос «что я узнаю и зачем»;
  - не повторяет дословно заголовок `title`;
  - не начинается с «В этом уроке...» / «In this lesson...» — описание должно сразу давать ответ.

  Плохо:
  > "In this lesson you will learn what a database is."

  Хорошо:
  > "A database is an organized collection of structured data managed by a DBMS. Learn types of databases, core DBMS functions, and why SQL is the universal data language."

- `keywords` — вспомогательный массив для внутренней разметки; не влияет напрямую на Google, но помогает внутренней системе тегирования курса.
- `teaches` — список учебных результатов для `LearningResource.teaches` в schema.org (что именно изучит читатель).
- `about` — список ключевых сущностей темы для `LearningResource.about` в schema.org (будет преобразован в массив `Thing`).

### Заголовок H1

Заголовок `#` не должен начинаться с `Lesson X.Y:` как единственного содержания — это «навигационный мусор» для поисковика.

Рекомендуемый формат: ключевой вопрос или утверждение + уточняющий подзаголовок:

```md
# What Is a Window Function? SQL Analytics Explained
```

Если нумерация урока важна для навигации, её можно вынести в frontmatter (`lesson: "1.1"`) или оставить как подпись, но не как первое слово H1.

### Заголовки H2 и H3

Поисковые системы читают `##` как структуру страницы. Хороший H2:

- отвечает на конкретный вопрос: `What Is a CTE in SQL?` лучше, чем `Definition`;
- содержит целевое ключевое слово или его вариацию;
- не превышает 60–70 символов.

Плохие примеры:

```
## Introduction
## Definition
## Why important?
```

Хорошие примеры:

```
## What Is a CTE (Common Table Expression)?
## Why Use CTEs Instead of Subqueries?
## CTE Syntax in PostgreSQL, MariaDB, and MySQL
```

### Вводный абзац (первые 150 символов = фактический snippet)

Google часто показывает первые строки страницы как описание в выдаче, если `<meta description>` не задан или признан нерелевантным.

Правило: **первый абзац урока должен прямо отвечать на вопрос**, а не «приветствовать читателя».

Нежелательно:
> "Welcome to an exciting journey into databases! In this lesson we'll lay the groundwork..."

Желательно:
> "A **CTE (Common Table Expression)** is a named temporary result set defined with `WITH` that makes complex queries easier to read and reuse. In this lesson you'll learn the syntax, key use cases, and how CTEs compare to subqueries."

### Alt-текст изображений

Шаблонный alt `"Lesson illustration"` не несёт смысла для поисковика. Alt должен описывать **содержание** иллюстрации:

```md
<!-- плохо -->
<img src="/images/lessons/lesson9_4-sql-view.svg" alt="Lesson illustration" width="100%">

<!-- хорошо -->
<img src="/images/lessons/lesson9_4-sql-view.svg" alt="Diagram showing how a SQL VIEW wraps a SELECT query and exposes it as a virtual table" width="100%">
```

### Внутренние ссылки

В конце каждого урока обязательно должна быть ссылка на следующий. Google использует внутренние ссылки для понимания структуры сайта и распределения веса:

```md
→ [Lesson 9.5: Updatable Views and INSTEAD OF Triggers](lesson9_5.en.md)
```

Если тема тесно связана с другим уроком курса — ссылайтесь на него по месту в тексте, не только в конце.

### FAQ-блок для featured snippets

Google активно показывает FAQ (блок «People Also Ask») в расширенных сниппетах. Добавьте блок в конце урока с 2–3 концептуальными вопросами:

```md
## Frequently Asked Questions

### What is the difference between a VIEW and a CTE?
A **VIEW** is stored in the database and reusable across sessions. A **CTE** exists only within a single query. Use a VIEW when the logic is shared across multiple queries; use a CTE for one-time query clarity.

### Can you nest CTEs?
Yes. You can define multiple CTEs in a single `WITH` clause and reference earlier ones from later ones in the same block.
```

В русской версии: `## Часто задаваемые вопросы`.

### Interview Questions — блок для поискового трафика

Запросы вида "SQL interview questions" или "database interview questions" имеют очень высокий объём поиска. Добавьте отдельный блок после FAQ с 3–5 вопросами в стиле технического собеседования:

```md
## Interview Questions

### How would you define a database in an interview?
A **database** is an organized, persistent collection of structured data managed by a DBMS. It allows multiple users and applications to store, retrieve, and manipulate data reliably and efficiently.

### What are the core responsibilities of a DBMS?
A DBMS handles data definition (schema), data manipulation (CRUD via SQL), transaction management (ACID), concurrency control, access security, backup/recovery, and data integrity constraints.
```

Правила для Interview Questions:
- Вопросы должны быть сформулированы так, как их задают на реальных собеседованиях.
- Ответы — сжатые, конкретные, с ключевыми терминами выделенными **жирным**.
- Ориентировочно 3–5 вопросов по теме урока.
- В русской версии: `## Вопросы для собеседования`.
- Размещать после FAQ, перед ссылкой на следующий урок.

### Время чтения

Добавьте строку с ориентировочным временем чтения сразу после frontmatter или в конце вводного абзаца:

```md
_Reading time: ~6 min_
```

Это пользовательский сигнал (снижает отказы) и хороший визуальный якорь.

### Ключевые слова в тексте

- Целевой ключевой запрос урока должен появиться в **первых 100 словах** текста.
- Используйте вариации: `CTE`, `common table expression`, `WITH clause` — а не только аббревиатуру.
- Не форсируйте вхождения: одно упоминание на 150–200 слов достаточно.

## Рекомендации для перевода на другие языки

При переводе уроков на English, Français и Português важно сохранять:

- ту же структуру разделов;
- тот же порядок примеров;
- тот же уровень детализации;
- ту же практическую направленность;
- те же SQL-блоки без изменения логики.

Можно адаптировать:

- естественные формулировки под язык;
- названия разделов;
- стилистические связки между абзацами.

Не нужно менять:

- смысл примеров;
- структуру урока;
- уровень сложности;
- последовательность объяснения.
