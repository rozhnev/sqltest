# Запустил зеркало SQL-сервиса в зоне .ru: как обойти блокировки и не дублировать тяжелый бэкенд

Когда часть аудитории начала писать «без VPN сервис не открывается», стало понятно: нужен отдельный вход для пользователей из РФ.

Я развиваю учебные SQL-платформы [sqltest.online](https://sqltest.online) и [sqlize.online](https://sqlize.online), где можно бесплатно тренироваться на реальных СУБД (PostgreSQL, Oracle, MariaDB 12.3, MS SQL 2025). Полный перенос инфраструктуры в РФ оказался дорогим, поэтому я сделал гибридную схему.

## Задача

Сохранить доступность сервиса для пользователей из РФ, не перенося ресурсоемкий контур баз данных.

## Решение в двух словах

- Фронтенд поднял в зоне .ru на российском хостинге.
- Бэкенд с Docker-контейнерами оставил в Германии.
- Связал их по server-to-server API.

Итог: сайт открывается стабильно, а тяжелые SQL-вычисления по-прежнему выполняются на мощной машине.

## Как устроена архитектура

### 1) Фронтенд в РФ

На стороне зеркала — нативный PHP + HTML/JS без фреймворков. Минимум зависимостей, проще деплой и поддержка.

```text
sqltest-online.ru (РФ)
├── PHP + HTML/JS
└── Точка входа для пользователей из РФ
```

### 2) Бэкенд в Германии

Основной сервер работает у Contabo: Docker Compose, несколько СУБД, большие образы, серьезная нагрузка.

```text
backend (Германия)
├── PostgreSQL
├── MariaDB
├── Oracle
└── другие контейнеры для SQL-песочниц
```

### 3) Поток запроса

1. Пользователь отправляет SQL-запрос на .ru-домен.
2. Фронт принимает запрос и делает вызов к бэкенду server-to-server.
3. Бэкенд выполняет SQL в нужной СУБД и возвращает результат.
4. Фронт отдает ответ пользователю.

## Почему это сработало

- Нет CORS-проблем: браузер работает только с .ru-доменом.
- Бэкенд можно закрыть по white-list IP.
- Интерфейс загружается локально для аудитории в РФ.
- Не пришлось дублировать дорогую инфраструктуру.

## Что пошло не так при деплое

После git pull всё выглядело нормально, но Composer зависал на установке зависимостей.

Причина: нестабильный доступ до packagist.org по маршруту с российского хостинга.

Проверяется быстро:

```bash
curl -v https://packagist.org
```

Если видите долгий таймаут, проблема может быть не в Composer, а в маршруте.

## Как починил

Хостер дал HTTP-прокси, и установка зависимостей сразу заработала:

```bash
export HTTPS_PROXY=http://proxy.host:port
composer install
```

Для Windows:

```bat
set HTTPS_PROXY=http://proxy.host:port
composer install
```

На практике это сэкономило много времени: вместо ручного дебага PHP/Composer проблема решилась на сетевом уровне.

## Выводы

1. Гибрид «фронт в РФ + бэкенд за рубежом» — рабочий компромисс для небольших продуктов.
2. Для бесплатного/edtech-проекта это намного дешевле, чем полный перенос.
3. В чек-лист деплоя в РФ стоит сразу добавить проверку доступности packagist.org.
4. Если Composer висит, первым делом проверяйте сеть и обсуждайте прокси с хостером.

Зеркало уже в работе: [sqltest-online.ru](https://sqltest-online.ru).

Если у вас был похожий опыт с зависимостями на российском хостинге, интересно сравнить подходы: прокси, локальные зеркала или private package registry.
